# 🎯 Как правильно просить AI о деплое на сервер

**Гайд для эффективной работы с AI ассистентом**

---

## 🤔 ЧТО ПОШЛО НЕ ТАК В ЭТОТ РАЗ?

### Проблема: AI не видел dist/ в репозитории

**Причина:**
```
dist/  ← эта папка была в .gitignore
```

Когда папка в `.gitignore`, Git **игнорирует её** - она не попадает в репозиторий.

Для AI это означало: **"dist/ не существует в проекте"**

### Что видел AI:

```
✅ src/               - исходный код (видел)
✅ package.json       - конфигурация (видел)
✅ README.md          - документация (видел)
❌ dist/              - НЕ ВИДЕЛ! (в .gitignore)
```

### Почему AI не догадался?

**Стандартная практика в разработке:**
- `dist/` (собранные файлы) **НЕ коммитят** в Git
- Их собирают **на сервере** или через CI/CD
- В репозитории хранят только **исходный код** (src/)

AI следовал этой практике автоматически.

## 💡 ПОЧЕМУ ЭТО ВАЖНО ПОНИМАТЬ?

### Что AI видит VS что ты видишь:

**Ты видишь на своём компьютере:**
```
onai-integrator-login/
├── src/           ✅
├── dist/          ✅ (есть на диске)
├── node_modules/  ✅ (есть на диске)
└── .env           ✅ (есть на диске)
```

**AI видит в Git репозитории:**
```
onai-integrator-login/
├── src/           ✅ (в Git)
├── dist/          ❌ (в .gitignore - не видит!)
├── node_modules/  ❌ (в .gitignore - не видит!)
└── .env           ❌ (в .gitignore - не видит!)
```

**Вывод:** AI работает с тем, что **закоммичено в Git**, а не с тем, что есть на твоём диске!

## ✅ КАК ПРАВИЛЬНО ПРОСИТЬ О ДЕПЛОЕ

### 🎯 ШАБЛОН #1: Полный деплой (всё автоматически)

**Правильная формулировка:**

```
Задеплой проект на Digital Ocean сервер полностью:
1. Пересобери dist/ локально из актуального src/
2. Убедись что dist/ попадёт в GitHub (проверь .gitignore)
3. Закоммить всё в Git и запушить в GitHub
4. Зайди на сервер Digital Ocean (178.128.203.40)
5. Подтяни обновления с GitHub (git pull)
6. Перезапусти Nginx

Сервер: root@178.128.203.40
Путь на сервере: /var/www/onai-integrator-login
Сайт: https://integratoronai.kz
```

**Что AI сделает:**
- ✅ Пересоберёт dist/ локально
- ✅ Проверит .gitignore
- ✅ Закоммитит и запушит в GitHub
- ✅ Зайдёт на сервер и обновит код
- ✅ Проверит что всё работает

---

### 🎯 ШАБЛОН #2: Только push в GitHub (без сервера)

**Правильная формулировка:**

```
Подготовь проект для деплоя:
1. Пересобери dist/ локально
2. Убедись что dist/ войдёт в коммит (не в .gitignore)
3. Закоммить всё и запушить в GitHub

Я сам потом обновлю сервер.
```

**Что AI сделает:**
- ✅ Пересоберёт dist/
- ✅ Проверит .gitignore
- ✅ Закоммитит и запушит
- ❌ НЕ будет трогать сервер (ты сказал что сам)

---

### 🎯 ШАБЛОН #3: Только обновление сервера (GitHub уже готов)

**Правильная формулировка:**

```
GitHub уже обновлён, теперь обнови сервер Digital Ocean:
1. Зайди на сервер 178.128.203.40
2. Перейди в /var/www/onai-integrator-login
3. Сделай git pull
4. Перезапусти Nginx
5. Проверь что сайт работает

Не трогай локальный код, не делай push - GitHub уже актуален.
```

**Что AI сделает:**
- ❌ НЕ будет трогать локальные файлы
- ❌ НЕ будет делать git push
- ✅ Зайдёт на сервер
- ✅ Подтянет обновления
- ✅ Перезапустит Nginx

## 🚨 КРИТИЧНЫЕ ДЕТАЛИ, КОТОРЫЕ НУЖНО УКАЗЫВАТЬ

### 1. **Где собирать dist/**

```
❌ Плохо: "Задеплой сайт"
✅ Хорошо: "Пересобери dist/ ЛОКАЛЬНО, потом запушь в GitHub"
```

**Почему важно:**
- AI может подумать что нужно собрать на сервере
- Или что dist/ уже готов
- Явное указание убирает путаницу

---

### 2. **Что делать с .gitignore**

```
❌ Плохо: "Залей всё в GitHub"
✅ Хорошо: "Убедись что dist/ НЕ в .gitignore, потом залей в GitHub"
```

**Почему важно:**
- Если dist/ в .gitignore, он не попадёт в GitHub
- AI должен знать что нужно проверить это

---

### 3. **Полный цикл или частичный**

```
❌ Плохо: "Обнови сайт"
✅ Хорошо: "Полный деплой: локально → GitHub → сервер → проверка"
```

**Почему важно:**
- "Обнови сайт" может значить что угодно
- Явное описание шагов гарантирует правильный результат

---

### 4. **Данные сервера**

```
❌ Плохо: "Обнови на сервере"
✅ Хорошо: "Обнови на сервере root@178.128.203.40 в папке /var/www/onai-integrator-login"
```

**Почему важно:**
- AI знает куда подключаться
- Не нужно гадать или искать в истории

## 🎓 ЧТО AI МОЖЕТ ДЕЛАТЬ АВТОМАТИЧЕСКИ

### ✅ AI МОЖЕТ:

1. **Читать любые файлы в проекте**
   - Исходный код (src/)
   - Конфигурации (package.json, .env, и т.д.)
   - Документацию

2. **Выполнять команды локально**
   - npm run build
   - git add/commit/push
   - Любые bash команды

3. **Подключаться к серверу по SSH**
   - ssh root@178.128.203.40
   - Выполнять команды на сервере
   - Проверять статус сервисов

4. **Работать с Git**
   - Коммитить изменения
   - Пушить в GitHub
   - Делать git pull на сервере

5. **Проверять результаты**
   - curl проверки
   - Статус Nginx
   - Логи сервера

---

### ❌ AI НЕ МОЖЕТ (или не должен):

1. **Видеть файлы в .gitignore**
   - Если файл игнорируется Git, AI его не видит в репозитории
   - Но может увидеть локально через file system

2. **Угадывать намерения**
   - "Обнови сайт" = неоднозначно
   - Лучше явно описать шаги

3. **Знать пароли/ключи**
   - Нужен доступ по SSH ключу
   - .env файлы нужно указывать явно

4. **Читать мысли 😄**
   - Чем подробнее запрос, тем лучше результат

## 📝 ИТОГ

### Что я понял из этой ситуации:

1. **dist/ был в .gitignore** → AI не видел его в Git
2. **AI следовал стандартной практике** → не коммитил билды
3. **Нужна была явная инструкция** → "убери dist из .gitignore и закоммить"

### Что ты должен делать в будущем:

1. ✅ **Описывай весь флоу деплоя** (локально → GitHub → сервер)
2. ✅ **Указывай данные сервера** (IP, путь, сайт)
3. ✅ **Уточняй про .gitignore** (должен ли dist/ попасть в Git)
4. ✅ **Проси проверку результата** (Last-Modified, curl, статус)

### Что AI будет делать:

1. ✅ Следовать твоим инструкциям **буквально**
2. ✅ Проверять .gitignore если ты попросишь
3. ✅ Делать полный цикл деплоя автоматически
4. ✅ Проверять результат и показывать отчёт
